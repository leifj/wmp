## What is WMP?

WMP (Wallet Messaging Protocol) is a multi-party messaging protocol built on
**JSON-RPC 2.0** with optional **MLS** (RFC 9420) end-to-end encryption.
It is designed for wallet-to-wallet, human-to-agent, and human-to-organization
communication. WMP is MCP-compatible â€” it shares the JSON-RPC 2.0 envelope
format with the Model Context Protocol.

## Key Concepts for Implementers

### Message Envelope

Every WMP message is a valid JSON-RPC 2.0 message. The `params` (or `result`)
object contains a `wmp` metadata object plus method-specific fields:

```json
{
  "jsonrpc": "2.0",
  "id": "msg-001",
  "method": "wmp.session.create",
  "params": {
    "wmp": { "version": "0.1", "sender": "did:web:alice.example.com" },
    ...method-specific fields
  }
}
```

### Methods

All methods use the `wmp.<domain>.<action>` naming convention:
- `wmp.session.create` / `wmp.session.close` â€” session lifecycle
- `wmp.message.deliver` / `wmp.message.ack` â€” message delivery
- `wmp.capability.update` / `wmp.capability.list` â€” capability negotiation
- `wmp.flow.start` / `.progress` / `.action` / `.complete` / `.error` â€” structured flows
- `wmp.resolve` â€” metadata resolution

### Capabilities

Capabilities are negotiated during session creation. Only negotiated
capabilities are usable. The standard capabilities are: `messaging`, `flows`,
`sign`, `mcp`, `relay`, `offline`, `resolve`. Profiles add more (e.g., `oid4vci`, `oid4vp`).

### Security Modes

| Mode | Description |
|------|-------------|
| `tls` | Transport-level TLS only (simplest) |
| `mls` | All application messages are MLS-encrypted |
| `mls-optional` | MLS available; `encrypted_capabilities` lists which caps require it |

### Conformance Levels

1. **WMP Basic** â€” JSON-RPC 2.0 + sessions + messages + TLS
2. **WMP Secure** â€” Basic + MLS encryption
3. **WMP Flows** â€” Basic + structured flows
4. **WMP Agent** â€” Basic + MCP capability
5. **WMP Registered** â€” Secure + evidence + signatures + timestamps
6. **WMP Wallet** â€” Flows + OpenID4x profile

### Error Codes

WMP error codes: -31000 (session not found) through -31012 (identity assertion
invalid). Standard JSON-RPC codes (-32700 to -32603) also apply.

## How to Implement

1. **Start with WMP Basic** â€” Parse/generate JSON-RPC 2.0, implement session
   create/close, message deliver, capability negotiation with TLS mode.

2. **Validate against schemas** â€” Use `schema/methods/*.json` to validate
   your message serialization. All schemas use JSON Schema Draft 2020-12.

3. **Run test vectors** â€” Load `vectors/*.json`. For each vector, send the
   `input` message and compare against `expected_response` or `expected_error`.
   Fields with value `"*"` match any valid value (server-generated IDs, etc.).

4. **Add conformance levels incrementally** â€” Each level adds a well-defined
   set of features. Implement in order: Basic â†’ Flows â†’ Secure â†’ Agent.

## Conventions

- All timestamps are ISO 8601 (`2026-04-29T10:15:30Z`)
- Session IDs match `^ses-[a-zA-Z0-9]+$`
- Message IDs should be globally unique (UUIDs recommended)
- Signatures use JCS (RFC 8785) canonicalization before signing
- Identifier schemes are listed in spec/wmp-core.md Section 7.2
